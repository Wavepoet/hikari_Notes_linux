# Volume 与 PV/PVC（存储）

容器的总是临时的，当容器被重启或删除时，容器的数据便会消失。和Docker类似，k8s也提供了Volume（数据卷）和PVC（Persistent Volume Claim，持久卷声明）来实现数据的持久化。

---

## Volume（卷）

Volume本质上是一个目录，存在于宿主机上，并且被挂载到容器中。

在k8s v1.18版本后的``kubectl run``中取消了``--volume``参数。因此无法使用命令的方式在创建Pod时挂载卷，但可以使用``kubectl set volume``去创建卷和挂载卷（不是很理解）。再加上现在使用k8s创Pod常用YAML。在此创建卷的方式也是使用的YAML文件的方式。

使用``kubectl apply -f <yaml文件>``提交YAML文件时，k8s会自动创建卷并挂载到Pod中。

### 常用Volume类型

- 临时性卷

  - **emptyDir**

    emptyDir可以翻译成空目录，当Pod被分配到Node上时创建，初始状态为空。Pod中的所有目录都可以读写emptyDir卷。如果容器崩溃重启，emptyDir中的数据会保留；但如果Pod被移出Node，数据将被永久删除。

    - **emptyDir的使用**

      ```yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: emptydir-share
        spec:
          containers:
            …………
            # 定义一个空的 emptyDir 卷
          volumes:
          - name: shared-storage
            emptyDir: {}
      ```

        基于内存的emptyDir卷：

      ```yaml
          volumes:
          - name: shared-storage
            emptyDir:
              medium: Memory # 定义一个基于内存的emptyDir卷
              sizeLimit: 500Mi #可选.限制这个空目录最多只能使用 500MB内存
      ```

        基于内存的emptyDir卷会将数据存储在RAM中，而不是磁盘上。这种类型的emptyDir卷适用于需要高速读写访问的场景，例如缓存数据或临时文件。需要注意的是，基于内存的emptyDir卷会占用Node的内存资源，因此在使用时要确保Node有足够的内存可用。

  - **hostPath**

    hostPath卷是把宿主机Node操作系统上的文件或目录挂载到Pod中。删除Pod的同时hostPath也会被删除。

    可以配置Tape来对hostPath进行检查：

    | Tape | 描述 |
    | --- | --- |
    | "" | 默认值，表示不检查 |
    | DirectoryOrCreate | 如果路径不存在就创建一个目录 |
    | Directory | 必须是一个目录 |
    | FileOrCreate | 如果路径不存在就创建一个文件 |
    | File | 必须是一个文件 |

    - **hostPath的使用**

      ```yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: hostpath-volume
        spec:
          containers:
          - name: test-container
            image: nginx
            volumeMounts:
            - mountPath: /container/logs # 容器内的路径
              name: log-volume
          volumes:
          - name: log-volume
            hostPath:
              path: /var/log/nginx     # 宿主机上的实际路径
              type: DirectoryOrCreate  # 如果没有就建一个
              type: Directory         # 必须是一个目录
      ```

- 持久性卷

  - **configMap**

    configMap卷是将K8中的ConfigMap对象数据挂载为文件或目录。
    ConfigMap是一种用于存储“配置数据”的核心资源对象，可以用ConfigMap来存储配置文件，命令行参数，环境变量等**非敏感数据**。

    - **configMap的使用**

      ```yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: configmap-volume
        spec:
          containers:
          - name: my-container
            image: nginx
            volumeMounts:
            - name: config-volume
              mountPath: /etc/config # 将 ConfigMap 卷挂载到容器内的 /etc/config 目录
          volumes:
          - name: config-volume
            configMap:
              name: my-configmap # 指定要挂载的 ConfigMap 对象的名称
      ```

        指定Key映射到特定路径:

      ```yaml
        volumes:
        - name: config-volume
          configMap:
            name: my-configmap # 指定要挂载的 ConfigMap 对象的名称
            items:
            - key: special-config # ConfigMap 中的键名
      ```

        只挂载单个文件：

      ```yaml
        volumeMounts:
        - name: config-volume
          mountPath: /etc/my-app.conf
          subPath: my-app.conf  # 只挂载这一个文件，不覆盖整个 /etc
      ```

        权限控制：

      ```yaml
        volumes:
        - name: config-volume
          configMap:
            name: my-configmap
            defaultMode: 0644 # 设置挂载文件的权限为 0644
      ```

  - **secret**

    和configMap卷类似，secret是将K8中的secret对象数据挂载为文件或目录。
    secret卷是k8s中**专门用来挂载敏感数据**的。Secret对象可以存储密码、OAuth令牌和ssh密钥等敏感信息。使用Secret对象可以避免在Pod定义中暴露敏感数据。

    - **secret的使用**

      将 Secret 中的某个 Key 映射为容器里的环境变量。

      ```yaml
      spec:
        containers:
        - name: secret-app
          env:
          - name: DB_PASSWORD
            valueFrom:
              secretKeyRef:
                name: my-db-secret  # Secret 的名字
                key: password       # Secret 里的键名
      ```

      作为卷挂载

      ```yaml
      spec:
        containers:
        - name: secret-app
          volumeMounts:
          - name: cert-volume
            mountPath: "/etc/certs"
            readOnly: true
        volumes:
        - name: cert-volume
          secret:
            secretName: my-tls-cert
      ```

      secret也支持挂载单个文件。挂载方式(语法)与ConfigMap一样。

      ImagePullSecrets是一种专门用于“拉取镜像”的特殊 Secret。当你私有镜像仓库，需要进行登录才能下载镜像时，就需要使用ImagePullSecrets来提供认证信息。

      ```yaml
      apiVersion: v1
      kind: Pod
      metadata:
        name: private-pod
      spec:
        containers:
        - name: my-app
          image: registry.cn-hangzhou.aliyuncs.com/my-repo/my-app:v1  # 私有镜像地址
          #-----------------------------
        imagePullSecrets:
        - name: <Secret名字>  # 这是一个已经创建好的Secret对象，里面存储了私有镜像仓库的认证信息
      ```

      可以使用kubecti来创建ImagePullSecrets：

      ```bash
      kubectl create secret docker-registry my-secret \
        --docker-server=<仓库地址> \
        --docker-username=<user> \
        --docker-password=<password> \
        --docker-email=<email>
      ```

  - **NFS**

    NFS卷是将**网络上的文件**挂载到Pod中。NFS是一种分布式文件系统协议，允许多个客户端通过网络访问共享的文件系统。

    - **NFS的使用**

      这是最基础的用法，直接在YAML中写死NFS服务器的地址。其他的用法在PV/PVC中会介绍。

      ```yaml
      apiVersion: v1
      kind: Pod
      metadata:
        name: nfs-pod
      spec:
        containers:
        - name: web
          image: nginx
          volumeMounts:
          - name: nfs-storage
            mountPath: /usr/share/nginx/html # 容器内的路径
        volumes:
        - name: nfs-storage
          nfs:
            server: 192.168.1.100  # NFS 服务器的 IP
            path: /data/nfs/public  # NFS 共享出的目录
            r eadOnly: false         # 是否只读
      ```

  - **projected**

    projected卷一种**复合类型的卷**，projected将多个现有的卷资源“映射”或“聚合”到同一个目录中。
    目前projected卷支持``secret``，``configMap``，``downwardAPI``,``serviceAccountToken``资源。

    - **projected的使用**

      ```yaml
      apiVersion: v1
      kind: Pod
      metadata:
        name: projected-volume
      spec:
        containers:
        - name: test-container
          image: busybox
          volumeMounts:
          - name: all-in-one
            mountPath: /app/config  # 所有的卷都会挂载在此目录
            readOnly: true
        volumes:  #配置卷
        - name: all-in-one
          projected: 
            sources:
            - configMap:  #configMap
                name: my-config
                items:
                  - key: config.yaml
                    path: settings.yaml # 投射后的文件名
            - secret:  #secret
                name: my-secret
                items:
                  - key: api-key
                    path: secret-key    # 投射后的文件名
            - downwardAPI:  #downwardAPI
                items:
                  - path: pod-name      # 投射后的文件名
                    fieldRef:
                      fieldPath: metadata.name
      ```

### 其他的Volume类型

- downwardAPI

用于注入Pod和容器的元数据。数据以纯文本格式的只读文件形式呈现。

- image

把容器镜像里的文件当作卷挂载。

- local

local卷是将本地存储设备，如磁盘， 分区或目录挂载到Pod中。

- iscsi

通过iSCSI协议网络挂载远程存储设备。

- fc(Fibre Channel，光纤通道)

允许将光纤通道 存储挂载到 Pod 中。

## 引入 PV / PVC

相较于其他的卷类型。PV和PVC才是实际生产环境中常用的卷类型。

- PV（Persistent Volume，持久卷）

PV是一个抽象出来的概念，是集群中的一块存储资源，它不属于任何命名空间，是全局共享的资源池。PV由管理员创建，或者通过动态供应器自动创建的。PV的生命周期独立于使用它的Pod。

或者说管理员或自动化程序会创建出来一堆的卷，这一个卷就是一个PV。

通常一个PV包含容量，访问权限，回收策略等参数。

- PVC(PersistentVolumeClaim,存储的使用请求)

PVC是用户对存储的请求，例如用户想要一个100GB的存储空间，那么K8s控制器会自动在现有的PV中寻找和100GB最接近的一个PV，为其分配给用户。

k8s寻找PV时原则是**大于或等于**PVC请求的大小，并且满足PVC的访问模式要求。

### 为什么实际生产环境使用PV/PVC

PV提供了一个抽象层，使得存储资源可以被动态地分配和管理，其的卷类型就比如NFS，在Pod中使用时需要写死NFS服务器的地址，换到另一个环境就必须修改Pod定义。

这同时带来一个问题，在配置其他Volume类型时，开发这需要知道底层存储的细节。在厂商的责任划分中，通常是云厂商才应该关注的事情。就算云厂商

PV/PVC中还有个StorageClass的技术，可以来动态地创建和管理存储资源，更好去做自动化。这个后面说。

### PV/PVC的使用

- 创建PV：

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv     # PV 的名称，在集群中必须唯一
spec:
  storageClassName: manual  
  capacity:
    storage: 10Gi   # 定义存储大小为 10GB
  volumeMode: Filesystem      # 指定卷的挂载模式，Filesystem为默认值，表示会被挂载为文件系统目录\
  accessModes:
    - ReadWriteOnce   # 该卷可以被单个节点以读写方式挂载
  persistentVolumeReclaimPolicy: Recycle  #定义 PV的回收策略，即当绑定的PVC被删除后，该PV该如何处理。Recycle (回收): 简单的清除卷上的基本数据，如执行 `rm -rf`，使其可以被再次绑定。注意: Recycle策略已被弃用，通常推荐Retain保留或Delete删除
  hostPath:
    path: /home/qyc/demo/my-pv  # 定义主机路径，即实际存储的位置
```

- 使用PVC：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  storageClassName: manual   # 寻找相同 class 的 PV
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 7Gi           # 申请 7Gi，会自动绑定到 10Gi 的 mysql-local-pv 上
```

- 虽后便可以使用这个PVC了：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
    - name: nginx-container
      image: nginx
      volumeMounts:
        - mountPath: "/usr/share/nginx/html" # 容器内的路径
          name: my-data-volume             # 必须对应下方的 volumes.name
  volumes:
    - name: my-data-volume
      persistentVolumeClaim:
        claimName: mysql-pvc               # 必须对应你刚才 get 出来的 PVC 名字
```

使用``kubectl get pv``和``kubectl get pvc``可以查看PV和PVC的状态。

```bash
qyc@qycLinux:~/k8s_hub$ kubectl get pv 
NAME    CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM               STORAGECLASS   VOLUMEATTRIBUTESCLASS   REASON   AGE
my-pv   10Gi       RWO            Recycle          Bound    default/mysql-pvc   manual         <unset>                          13s
qyc@qycLinux:~/k8s_hub$ kubectl get pvc
NAME        STATUS   VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE
my-pvc   Bound    my-pv    10Gi       RWO            manual         <unset>                 21s
qyc@qycLinux:~/k8s_hub$ ^C
```

## 动态卷供应

在实际生产中，管理员不可能提前创建好一大堆的PV来满足用户的需求。k8s提供了动态卷供应的功能，可以根据PVC的请求自动创建PV。

### StorageClass（SC,存储类）

StorageClass是一个存储模板，当用户使用PVC申请PV时，系统会根据StorageClass的定义自动创建相应的PV。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-dynamic-storage     #存储类的名称
  annotations:
    storageclass.kubernetes.io/is-default-class: "true" #设置为默认类，k8s会默认使用这个StorageClass来动态创建PV
provisioner: k8s-sigs.io/nfs-subdir-external-provisioner #存储提供者
parameters:     #后端特有参数，这与实际使用的插件相关，插件不同则不同。我使用的是NFS的插件
  archiveOnDelete: "false"     #不归档数据，直接删除
reclaimPolicy: Retain           #回收策略
allowVolumeExpansion: true          #允许扩容
volumeBindingMode: Immediate          #绑定模式
mountOptions:      #挂载选项
  - hard   #网络断开时请求会一直重试
  - nfsvers=4.1
```

- provisioner

用于标识存储的提供者，在这里我使用的是本地的NFS。这个的前提是你的本地得搭有NSA服务器。如果服务器在云上，可以使用云厂商门的云存储服务。云厂商们的Provisioner都不同，请以实际为准。还可以使用一些私有云存储。

## 网络存储

### nfs-subdir-external-provisioner

NFS本身并不支持动态分配。现在主流使用的是``nfs-subdir-external-provisioner``这一插件来管理NFS存储与PV/PVC之间的关系。他可以在已有的NFS共享目录中，自动为每一个PVC创建一个子目录，并将其转化为 PV。

在k8s中跑一个专门的``nfs provisioner pod``来监听PVC请求，并自动构架目录来满足PVC请求。

- RBAC权限配置

因为nfs provisioner需要监听PVC请求，管理PV。所以他需要拥有操作API的权限。

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: nfs-client-provisioner
  namespace: default
---
kind: ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: nfs-client-provisioner-runner
rules:
  - apiGroups: [""]     # 核心 API 组
    resources: ["nodes"]      # 资源对象：PV
    verbs: ["get", "list", "watch"]     # 动作：创建、删除
  - apiGroups: [""]
    resources: ["persistentvolumes"]
    verbs: ["get", "list", "watch", "create", "delete"]
  - apiGroups: [""]
    resources: ["persistentvolumeclaims"]
    verbs: ["get", "list", "watch", "update"]
  - apiGroups: ["storage.k8s.io"]
    resources: ["storageclasses"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["create", "update", "patch"]
---
kind: ClusterRoleBinding
apiVersion: rbac.authorization.k8s.io/v1      # 指向身份证
metadata:
  name: run-nfs-client-provisioner      # 指向权限清单
subjects:
  - kind: ServiceAccount
    name: nfs-client-provisioner
    namespace: default
roleRef:
  kind: ClusterRole
  name: nfs-client-provisioner-runner
  apiGroup: rbac.authorization.k8s.io
```

部署 nfs Provisioner pod

```yaml
kind: Deployment
apiVersion: apps/v1
metadata:
  name: nfs-client-provisioner
  namespace: default
spec:
  replicas: 1
  strategy:
    type: Recreate
  selector:
    matchLabels:
      app: nfs-client-provisioner
  template:
    metadata:
      labels:
        app: nfs-client-provisioner
    spec:
      serviceAccountName: nfs-client-provisioner
      containers:
        - name: nfs-client-provisioner
          image: registry.k8s.io/sig-storage/nfs-subdir-external-provisioner:v4.0.2
          volumeMounts:
            - name: nfs-client-root
              mountPath: /persistentvolumes
          env:
            - name: PROVISIONER_NAME
              value: k8s-sigs.io/nfs-subdir-external-provisioner # 必须与StorageClass中的一致
            - name: NFS_SERVER
              value: 172.16.128.196  # NFS服务器的IP
            - name: NFS_PATH
              value: /nfs/data      # NFS共享目录
      volumes:
        - name: nfs-client-root
          nfs:
            server: 172.16.128.196  # NFS服务器的IP
            path: /nfs/data
```

### Ceph/Rook

这是企业自建服务器通常使用的方案。

- Ceph

Ceph是一款开源的软件定义存储平台（SDS），可以提供块存储，对象存储，文件存储三种存储方式。

- Rook

Rook 是一个云原生存储编排器。它利用了 K8s 的 Operator（自定义控制器）模式，将 Ceph 的运维逻辑编写成了代码。

配置过程：

- 下载Rook

```bash
wget https://github.com/rook/rook/archive/refs/tags/v1.13.10.tar.gz
tar xf rook-1.13.10.tar.gz
cd rook-1.13.10/deploy/examples
```

- 部署 Rook Operator

```bash
kubectl create -f crds.yaml
kubectl create -f common.yaml
kubectl create -f operator.yaml
```

- 部署 Ceph 集群

创建并编辑YML文件。

```yaml
apiVersion: ceph.rook.io/v1
kind: CephCluster
metadata:
  name: rook-ceph
  namespace: rook-ceph
spec:
  cephVersion:
    image: quay.io/ceph/ceph:v17.2.6          # Ceph 版本
    allowUnsupported: false
  dataDirHostPath: /var/lib/rook               # 宿主机数据存储路径
  mon:
    count: 3                                   # Monitor 数量（推荐奇数）
    allowMultiplePerNode: false
  mgr:
    count: 2
  dashboard:
    enabled: true                              # 启用 Ceph Dashboard
    ssl: true
  storage:
    useAllNodes: true                          # 使用所有节点
    useAllDevices: true                        # 使用所有可用设备[reference:15]
    deviceFilter: "^sd[b-z]"                   # 过滤指定设备
  crashCollector:
    disable: false
```

```bash
kubectl create -f <YML文件>
```

进入 Toolbox 执行 Ceph 命令

```bash
kubectl exec -it -n rook-ceph $(kubectl get pod -n rook-ceph -l "app=rook-ceph-tools" -o jsonpath='{.items[0].metadata.name}') -- bash

#执行
ceph -s
ceph osd status
```

- 启用 Ceph Dashboard

Rook支持Dashboard(web)，可开可不开。下面是步骤。

部署对外访问的 Dashboard

```bash
kubectl apply -f dashboard-external-https.yaml
```

```bash
kubectl get svc -n rook-ceph
```

获取登录密码

```bash
kubectl -n rook-ceph get secrets rook-ceph-dashboard-password -o jsonpath='{.data.password}' | base64 -d ; echo
```

访问``https://<任意节点IP>:<dashboard_port>``节点登录即可。

- 验证

创建CephBlocPool

```yaml
apiVersion: ceph.rook.io/v1
kind: CephBlockPool
metadata:
  name: replicapool
  namespace: rook-ceph
spec:
  failureDomain: host                    # 故障域级别
  replicated:                            # 副本模式
    size: 3                              # 副本数（默认 3）
  parameters:
    pg_num: 128                          # 归置组数量
    pgp_num: 128                         # PGP 数量
```

创建``storageClass``供PVC调用

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ceph-block
provisioner: rook-ceph.rbd.csi.ceph.com
parameters:
  clusterID: rook-ceph
  pool: replicapool
  imageFormat: "2"
  imageFeatures: layering
  csi.storage.k8s.io/node-stage-secret-name: rook-csi-rbd-node
  csi.storage.k8s.io/node-stage-secret-namespace: rook-ceph
  csi.storage.k8s.io/provisioner-secret-name: rook-csi-rbd-provisioner
  csi.storage.k8s.io/provisioner-secret-namespace: rook-ceph
reclaimPolicy: Delete
```

使用PVC挂载存储

```bash
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-pvc
spec:
  storageClassName: ceph-block
  accessModes:
    - ReadWriteOnce    # 块存储仅支持 RWO
  resources:
    requests:
      storage: 10Gi
```
